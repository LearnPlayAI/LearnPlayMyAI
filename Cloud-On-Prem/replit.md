# Overview
LearnPlay is a production-ready, gamified education platform offering AI-powered lesson creation, an e-learning marketplace, and enterprise license management. It provides a secure, customizable, and gamified learning solution with zero-hallucination AI content, supporting both cloud and self-contained on-premises deployments from a single codebase.

# User Preferences
Preferred communication style: Simple, everyday language.

**Database & Architecture Policy:**
- Before creating or altering any tables/columns for features, FIRST check existing database tables that can be reused
- MUST follow single source of truth architecture - never duplicate data storage
- For ALL issues identified or reported, MUST find root causes - no fallbacks or workarounds
- System must be robust and consistent
- **CRITICAL: All database schema changes must be done MANUALLY via SQL - NEVER use `npm db:push` as it runs into interactive waiting for input issues that hang the process**
- **CRITICAL: Every database schema change (new table, new column, altered column, new index, etc.) MUST include a corresponding SQL migration file in `migrations/` for on-prem deployments. Use `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` for safety. Number sequentially after the latest file (e.g., 0054_xxx.sql). This applies to ALL schema changes — no exceptions.**
- **CRITICAL: When adding a new migration SQL file, you MUST ALSO add a corresponding entry to `migrations/meta/_journal.json` with the next sequential `idx`, `tag` matching the filename (without .sql), and a `when` timestamp. Drizzle's migrator will not recognize the file without this journal entry.**

**AI Content Generation Policy:**
- For ANY AI-generated content, prevent AI hallucinations as best possible
- Always use source document text as the primary source of truth
- Validate generated content against source material using validateAgainstSource()
- Include zero-hallucination constraints in all AI prompts
- Track source citations with sourceMap for content provenance

**Cloud / On-Prem Compatibility Policy:**
- When building ANY new feature or fixing ANY bug on the cloud version, ALWAYS consider the on-prem impact
- Any change to shared server code (routes, services, middleware) MUST work in both cloud and on-prem modes
- On-prem-only features must NEVER be applied to the cloud version
- Cloud-only features must NEVER leak into on-prem builds
- When modifying any of the 4 swapped files (`db.ts`, `migrate.ts`, `objectStorage.ts`, `vite.ts`), the corresponding on-prem variant MUST also be updated if the change is relevant
- Test both code paths when adding environment-gated behavior (e.g., `ONPREM_MODE`, `ONPREM_OWN_API_KEYS`)
- **CRITICAL: ALL on-prem script changes MUST be made to `onprem/` source files — NEVER edit `dist-onprem/` directly. `dist-onprem/` is generated by `onprem/build-onprem.sh` and will be overwritten on every build. The build script copies scripts from `onprem/` into `dist-onprem/scripts/`.**

**Base URL Construction Policy:**
- ALL user-facing URLs MUST use `getBaseUrl()` from `server/config/base-url.ts` — NEVER construct URLs inline from process.env
- `getBaseUrl()` reads env vars dynamically per call (BASE_URL → PUBLIC_BASE_URL → FRONTEND_URL → VITE_DOMAIN → REPLIT_DEV_DOMAIN → localhost fallback)
- NEVER cache `getBaseUrl()` in module-level constants, constructor properties, or static class fields — always call it at the point of use
- `getEmailFrom()` from same module provides centralized email sender address
- On-prem `ssl-mode.sh` update_env() must update ALL 5 URL env vars: BASE_URL, FRONTEND_URL, VITE_DOMAIN, PUBLIC_BASE_URL, COOKIE_SECURE

# System Architecture

## UI/UX Decisions
The frontend uses React with TypeScript, Shadcn/ui (Radix UI), and Tailwind CSS, featuring a dark mode casino theme, mobile-first design, and white-label branding. It ensures WCAG AA compliance with an auto-correcting, theme-aware contrast system.

## Technical Implementations
- **Frontend**: React (Vite), Wouter for routing, TanStack Query for data fetching, React Hook Form with Zod for forms.
- **Backend**: Express.js with TypeScript and Node.js for RESTful APIs.
- **Data Storage**: PostgreSQL with Drizzle ORM, ensuring tenant isolation.
- **Authentication & Security**: Express sessions, Zod validation, HMAC SHA256, audit trails, secure password reset, multi-layered tenant isolation.
- **Monetization**: Multi-currency, LP Credits System with organization-level wallets.
- **Performance**: Database pooling, composite indexes, observability, in-memory rate limiting, N+1 query remediation, object storage caching.
- **System Integrations**: Payment Orchestrator, unified email transport, centralized feature flags.
- **AI Features**: AI Feedback and Fix systems, "Zero auto-retries" policy for lesson generation, multi-language support (29 languages) using Gemini, with translation wizards.
- **On-Premises Deployment**: Self-contained system with build-time file swapping, automated installation scripts, a comprehensive administration CLI (`lpadmin.sh`), and full disaster recovery (DR) backup/restore.
- **Backup System**: Centralized backup directory (`/lppbackups`), daily DB backups, WAL archiving for PITR, full-system DR archives.
- **Disaster Recovery**: `dr-backup.sh` creates archives with 3 modes: Clone (exact replica), Create-Prod (exclude transactional data), System-Copy (data refresh preserving target identity). `dr-restore.sh` restores with mode-aware behavior. Per-customer encryption keys from provision bundles with legacy passphrase fallback.

## Feature Specifications
- **Gamified Learning**: Battle card games, XP, leveling, leaderboards, powerups, cosmetics, inventory, season passes.
- **Organization Management**: Single-org membership, 3-level hierarchy, central management hub, SuperAdmin dashboard.
- **E-Learning Marketplace**: Course creation, purchasing, enrollment, reviews, multi-currency, monthly payouts.
- **AI-Powered Content**: Google Gemini for AI Quiz, Lesson, and Topic Generation; Gamma.app API for lesson generation from documents. AI Content Coach with quality rubric and Bloom's taxonomy tracking, enforcing zero-hallucination.
- **Lesson & Course Creation**: Auto-generates overviews and key takeaways using Gemini AI, with readiness checks and document uploads (Word, PPTX, video). Guided "Next Step" workflow on Course Lessons page with per-lesson step tracking, course-level gating, on-prem video requirement enforcement, detailed progress dashboard, and pre-publish checklist.
- **Reporting & Analytics**: Learner reporting, e-learning sales reports, SuperAdmin payout management, LP Credits analytics, Org Admin Sales Dashboard with ROI metrics and detailed enrollment table.
- **Access Control**: Public course certificates, purchaser access override, course card organization branding, public showcase, cross-org assignment. Commission Rate hidden from CustSuper. Sales Dashboard restricted to SuperAdmin/CustSuper with org filter dropdown and impersonation auto-select.
- **Central Management Hub (CMH) Cross-Org Course Assignment** (on-prem only): Partner organizations appear inline in the CMH hierarchy tree as top-level "Organization" nodes with amber styling and "Partner" badge. Partner nodes are read-only (no Add/Edit/Delete/Assign Users). Each partner org expands to show its full hierarchy (depts → units → teams). Course assignment (assign/edit due date+mandatory/remove) works for both own-org and partner-org nodes. Cross-org assignments use public courses only. Independent assignments: same course can be assigned to multiple scopes across orgs with different due dates and mandatory values. Backend endpoints: GET /api/interorg/my-public-courses, GET /api/interorg/target-orgs/:orgId/scope-courses, GET /api/courses/org-courses. All gated behind ONPREM_MODE + isTeacherOrAdmin.
- **Enterprise Customer Portal** (cloud only): Separate enterprise login/registration system for B2B on-prem customers. Features: company info management, sub-company hierarchy, document uploads (business registration, banking, address proofs), build version downloads, license request/key management, agreement templates. Routes: /api/enterprise/auth/*, /api/enterprise/profile, /api/enterprise/sub-companies, /api/enterprise/documents, /api/enterprise/builds, /api/enterprise/licenses, /api/enterprise/agreements, /api/enterprise/revenue/*. SuperAdmin management at /api/admin/enterprise/*. Frontend at /enterprise/*.
- **On-Prem Production Licensing**: EC P-256 asymmetric crypto. License keys are ECDSA-signed (not encrypted) for authenticity/integrity verification. License requests are ECDH+AES-256-GCM hybrid encrypted for confidentiality. Cloud private key (CLOUD_LICENSE_PRIVATE_KEY) stays cloud-only; public key bundled into on-prem builds at server/config/cloud-license-public-key.pem. CustSuperAdmin generates hardware key (SHA-256 of hostname) + encrypted license request file via /api/onprem/license/generate-request. Cloud SuperAdmin decrypts .lreq files, sets monthly fee, approves, generates signed license key (12-month expiry). On-prem validates locally (ECDSA signature + hardware key match + expiry check, no call-home). Tables: enterprise_customers, enterprise_documents, build_versions, enterprise_license_requests, enterprise_license_keys, onprem_license_state, enterprise_revenue_sync, enterprise_agreement_templates, enterprise_keyring.
- **Per-Customer Key Management** (cloud only): AES-256 symmetric keys per customer for backup encryption, secrets management, and vault operations. Keys stored in enterprise_keyring table, encrypted at rest using HKDF-derived master key from CLOUD_LICENSE_PRIVATE_KEY. Key types: backup, secrets, vault. Auto-provisioned on license approval. SuperAdmin endpoints: decrypt .lreq, list/provision/rotate/bundle customer keys. Customer endpoints: list keys, download provision bundle. Provision bundles contain decrypted key material (base64) signed with ECDSA. On-prem scripts read keys from /opt/learnplay/keys/provision-bundle.json.
- **Learner Registration Gating** (on-prem only): Registration route checks onprem_license_state for valid production license before allowing learner registration. Non-learner roles (custsuper, orgadmin, teacher) are not gated.

## System Design Choices
- **Routes Architecture**: Modular, domain-specific organization.
- **Business Package Subscription System**: Dedicated database tables for eligibility, billing, and seat enforcement.
- **Zero-Hallucination & Content Quality**: AI content validation, source citation tracking, content health scoring, AI-powered remediation.
- **Cloud vs. On-Prem Architecture**: Single codebase for cloud (Replit, Neon, GCS) and on-premises (Nginx, PM2, local Postgres, local filesystem) via build-time file swapping.
- **On-Prem API Keys**: Customers must always provide their own API keys (Gemini, Gamma, MailerSend if not using SMTP). Payment gateway disabled for on-prem; credits managed by CustSuper admin.
- **Encryption-Aware Service Control**: Services manage encrypted `.env.enc` for PM2 startup and configuration changes.
- **Payment Gateway Gating**: `PAYMENT_GATEWAY_ENABLED` feature flag controls Yoco payment routes and frontend UI visibility.
